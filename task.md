## Этапы сквозного проекта

Для сквозного проекта можно выделить следующие этапы:

### Этап 1: Базовая работа с «железками»

**Цель: увидеть жизнь во всех компонентах**

- Каждый датчик по отдельности работает
- Простой вывод в Serial Monitor

**Результат этапа: что-то мигает, пищит и показывает цифры**

### Этап 2: Некоторая автономность «железок»

**Цель: автономное устройство с логикой**

- Сбор данных со всех датчиков
- Локальная обработка (фильтрация, усреднение)
- Простая логика: движение → включить свет

**Результат этапа: устройство само решает, когда пищать**

### Этап 3: «Железки» в простейшей сети

**Цель: устройство в сети**

- Подключение к Wi-Fi
- Отправка данных на MQTT брокер или аналог хоть самописный
- Получение команд извне

**Результат: вижу температуру на телефоне с другого конца кампуса**

### Этап 4: «Железки» в «умной» сети

**Цель: интеллект в облаке**

- База данных временных рядов (InfluxDB или аналог хоть самописный)
- Правила и автоматы (Node-RED или аналог хоть самописный)
- Визуализация (Grafana или аналог хоть самописный)

**Результат: графики, уведомления, история за месяц**

### Этап 5: Бизнес-ценность «железки» в «умной» сети

**Цель: Система приносит пользу**

- REST API для интеграции
- Приложение для телефона / Telegram-бот для управления
- Бизнес-логика: анализ трендов, предсказание отказов, интеграция с внешними сервисами

**Результат: система взаимодействуют с внешним миром, обладает сложной аналитикой**

## Требования к архитектуре

При этом важно строить полноценную «взрослую» архитектуру со следующими принципами:

- Многоуровневость. Выделить:
- Слой драйверов (работа с железом)
- Слой сервисов (обработка данных)
- Слой коммуникации (MQTT/HTTP)
- Слой бизнес-логики (правила)
- Отказоустойчивость. Пример в коде:

```c
// НЕ ТАК:
void loop() {
  temp = readDHT(); // если датчик сломался - всё стоп
}

// ТАК:
void loop() {
  if (millis() - lastRead > 1000) {
    ErrorCode err = readSensor();
    if (err == OK) {
      processData();
    } else {
      logError(err);
      useLastGoodValue(); // система работает даже при сбоях
    }
  }
}
```
- Масштабируемость. Вы спроектируете систему так, что завтра можно:
- Добавить ещё 10 датчиков (через I2C)
- Подключить ещё 5 таких же устройств
- Развернуть на 100 таких устройствах без переписывания
- Безопасность. Вы реализуете:
- Авторизацию на MQTT (логин/пароль)
- Шифрование данных
- Валидацию входящих команд
- Наблюдаемость. Система будет сообщать о себе:
- Метрики
- Логи: что происходит, ошибки, предупреждения
- Трейсы: путь данных от сенсора до облака
- Управляемость. Вы сможете:
- Изменить конфигурацию без перепрошивки
- Диагностировать проблемы удалённо
- Экономическая эффективность. Вы научитесь считать:
- Энергопотребление в разных режимах
- TCO (Total Cost of Ownership) вашего решения

## Пример проекта от «железок» до «верхнего» уровня

### Умная система контроля микроклимата и доступа

### Часть 1: Исходная проблема и концепция

#### Контекст: на кафедре микроэлектроники университета находится лаборатория с дорогостоящим оборудованием: микроскопами, зондовыми станциями, анализаторами. Это оборудование требует строгого соблюдения климатических условий (температура 20-24°C, влажность 40-60%) и ограниченного доступа в помещение. Исторически сложилось, что контроль осуществлялся вручную: дежурный лаборант раз в час записывал показания термометра и гигрометра в журнал, доступ регулировался обычным замком с ключами.

#### Проблемы, с которыми столкнулись:

- За три года вышли из строя два микроскопа из-за перепадов температуры и влажности
- Были случаи несанкционированного доступа в лабораторию в нерабочее время
- При возникновении проблем (например, отключении кондиционера) реакция занимала несколько часов — пока кто-то не зайдёт в помещение
- Аудит использования оборудования был практически невозможен

#### Бизнес-требования от заказчика (заведующего кафедрой):

- Снизить риск поломки оборудования минимум
- Обеспечить контроль доступа с ведением журнала
- Сократить время реакции на отклонения климатических параметров до 15 минут
- Предоставить возможность удаленного мониторинга
- Интегрироваться с существующей системой безопасности университета
- Уложиться в бюджет 15 000 рублей на аппаратную часть

#### Технические требования:

- Точность измерения температуры: ±0.5°C
- Точность измерения влажности: ±3%
- Время отклика системы на события: менее 5 секунд
- Автономная работа при обрыве связи: не менее 24 часов
- Хранение исторических данных: 1 год
- Гарантированная доставка критических уведомлений

### Часть 2: Архитектурное проектирование системы

### 2.1. Аппаратная архитектура

Система была спроектирована как распределённая, состоящая из двух типов узлов, которые могут быть объединены в 1 узел при необходимости:

#### Узел 1: Климатическая станция (размещается внутри лаборатории)

- Микроконтроллер NodeMCU v3 Lolin с Wi-Fi модулем ESP8266
- Датчик температуры и влажности DHT22, размещённый в термоэкранированном корпусе
- Инфракрасный датчик движения для обнаружения присутствия людей
- Модуль дисплея TM1638 для локального отображения параметров
- Зуммер для звуковой сигнализации
- RGB светодиод для визуальной индикации состояния
- Резервное питание от power bank на 10 000 мАч

#### Узел 2: Контроллер доступа (размещается у входа в лабораторию)

- Микроконтроллер NodeMCU v3 Lolin
- Матричная клавиатура 3×4 для ввода PIN-кодов
- Модуль дисплея TM1638 для отображения статуса доступа
- Сервопривод Tower Pro 360, механически соединённый с защёлкой двери
- Инфракрасный датчик движения для детекции приближения к двери
- Модуль расширения I2C с дополнительными портами

#### Связь между узлами: Оба узла соединяются по Wi-Fi через локальную сеть университета, общаются через MQTT-брокер.

### 2.2. Программная архитектура

#### Прошивка микроконтроллеров построена по модульному принципу:

- Драйверный слой: низкоуровневые функции работы с аппаратными компонентами
- Сервисный слой: обработка данных, фильтрация, усреднение
- Сетевой слой: управление подключением, MQTT-клиент, переподключение
- Бизнес-логика: локальные правила и реакции

#### Серверная часть развёрнута на виртуальной машине в дата-центре университета:

- MQTT-брокер Mosquitto для обмена сообщениями
- База данных временных рядов InfluxDB для хранения показаний датчиков
- Реляционная база данных PostgreSQL для структурированных данных
- Микросервис на Python (FastAPI) для бизнес-логики
- Node-RED для визуального программирования автоматизации
- Grafana для визуализации и дашбордов
- Nginx как обратный прокси и веб-сервер

### 2.3. Сетевая архитектура

#### Все компоненты объединены в единую сетевую инфраструктуру:

- Устройства подключаются к Wi-Fi сети университета в отдельной VLAN
- MQTT-трафик шифруется с помощью TLS
- REST API защищён с помощью JWT-токенов
- Используется статическая маршрутизация с приоритетами
- Реализован мониторинг сетевого трафика и качества связи

### Часть 3: Поэтапная реализация проекта

### Этап 1: Прототипирование

На этом этапе команда сфокусировалась на создании рабочего прототипа. Были решены следующие задачи:

#### Задача 1.1: Аппаратная сборка

- Собраны два физических устройства на монтажных платах
- Проведена калибровка датчиков DHT22 по эталонному гигрометру
- Настроена механика сервопривода для управления защёлкой двери
- Реализовано экранирование датчиков от помех

#### Задача 1.2: Базовая прошивка

- Реализован сбор данных с датчиков каждые 30 секунд
- Настроена фильтрация показаний (скользящее среднее за 5 измерений)
- Реализована индикация состояния через RGB светодиод:
- Зелёный: норма
- Жёлтый: предупреждение
- Красный: тревога
- Настроен дисплей TM1638 для отображения температуры и влажности

#### Задача 1.3: Локальная логика

- Реализованы простые правила:
- Если температура выше 24°C более 5 минут → желтый светодиод + звуковой сигнал
- Если движение обнаружено в нерабочее время → красный светодиод + запись события
- Ввод правильного PIN-кода → открытие двери на 10 секунд

**Результат этапа: Два физических устройства, самостоятельно работающих по заданной логике, выводящих данные на дисплеи и реагирующих на изменения.**

### Этап 2: Сетевая интеграция

На этом этапе устройства были подключены к сети и начали обмениваться данными с сервером.

#### Задача 2.1: Настройка MQTT-инфраструктуры

- Установлен и настроен MQTT-брокер Mosquitto
- Созданы раздельные топики для разных типов данных:
- lab/climate/temperature — показания температуры
- lab/climate/humidity — показания влажности
- lab/access/events — события доступа
- lab/access/commands — команды управления
- Реализована аутентификация устройств по логину и паролю
- Настроено сохранение сообщений (retain) для статусных топиков

#### Задача 2.2: Реализация MQTT-клиента на устройствах

- Реализован стабильный MQTT-клиент с автоподключением
- Добавлена буферизация сообщений при потере связи
- Реализованы QoS уровни для гарантированной доставки критических сообщений
- Настроен Last Will and Testament для отслеживания состояния устройств

#### Задача 2.3: Первичная обработка данных на сервере

- Создан Python-сервис, подписанный на MQTT-топики
- Реализована запись данных в InfluxDB с тегами:
- device_id: идентификатор устройства
- location: местоположение (lab_inside/door)
- sensor_type: тип датчика
- Настроена агрегация данных (среднее за 1, 5, 60 минут)

#### Задача 2.4: Локальный веб-интерфейс

- Создан простой веб-интерфейс на Flask для мониторинга
- Реализованы графики температуры и влажности за 24 часа
- Добавлена таблица последних событий доступа
- Реализованы кнопки ручного управления дверью

**Результат этапа: Устройства стабильно передают данные в облако, данные сохраняются в базу, доступен веб-интерфейс для мониторинга.**

### Этап 3: Облачная логика и автоматизация

На этом этапе была реализована интеллектуальная обработка данных и автоматические реакции.

#### Задача 3.1: Настройка бизнес-правил в Node-RED

- Создан поток обработки климатических данных:
- Если температура > 25°C более 10 минут → отправка Telegram-уведомления дежурному
- Если влажность < 35% или > 65% → включение увлажнителя/осушителя через smart-розетку
- Если температура растёт быстрее 0.5°C в минуту → тревога о возможной поломке кондиционера
- Создан поток обработки событий доступа:
- Если попытка доступа в нерабочее время → запрос подтверждения у администратора
- Если 3 неудачные попытки ввода PIN → блокировка на 15 минут и уведомление охране
- Если дверь открыта более 5 минут → звуковое предупреждение

#### Задача 3.2: Интеграция с внешними системами

- Реализован Telegram-бот с командами:
- /status — текущие показания
- /history N — история за N часов
- /open_door — дистанционное открытие двери (только для администраторов)
- /report — ежедневный отчёт
- Интеграция с Google Календарём для учёта рабочих часов и мероприятий
- Отправка email-уведомлений для критических событий

#### Задача 3.3: Реализация сложной бизнес-логики

- Алгоритм прогнозирования температуры на основе исторических данных
- Адаптивные пороги срабатывания (учёт сезонности, времени суток)
- Система оценки рисков поломки оборудования на основе совокупности факторов
- Алгоритм обнаружения аномалий (например, внезапное падение влажности)

#### Задача 3.4: Настройка базы данных

- InfluxDB: хранение показаний датчиков с частотой 30 секунд (со сжатием старых данных)
- PostgreSQL: структурированные данные:
- Таблица users — пользователи и их права
- Таблица access_log — журнал доступа
- Таблица alarms — история аварийных событий
- Таблица device_status — статусы устройств
- Таблица maintenance — журнал обслуживания

**Результат этапа: Система автоматически реагирует на события, отправляет уведомления, интегрирована с внешними сервисами.**

### Этап 4: Пользовательские интерфейсы и отчётность

На этом этапе были созданы удобные интерфейсы для разных категорий пользователей.

#### Задача 4.1: Оперативный дашборд в Grafana

- Созданы панели:
- Температура и влажность в реальном времени с допустимыми диапазонами
- Тепловая карта температуры за сутки
- Статусная панель устройств (онлайн/офлайн, батарея, сигнал Wi-Fi)
- Счётчики событий за сегодня (успешные/неуспешные попытки доступа)
- Лента событий в реальном времени
- Настроены алерты в Grafana для визуального привлечения внимания
- Реализовано разделение прав доступа к дашбордам

#### Задача 4.2: Аналитическое веб-приложение

- Разработано SPA-приложение на Vue.js:
- Отчёты за произвольные периоды с фильтрами
- Анализ трендов температуры и влажности
- Статистика использования лаборатории по времени, пользователям
- Оценка энергоэффективности (если бы были данные по энергопотреблению)
- Экспорт данных в PDF и Excel
- Реализована система ролей:
- Лаборант: только просмотр текущих показаний
- Преподаватель: доступ к истории своего оборудования
- Администратор: полный доступ, управление пользователями

#### Задача 4.3: Мобильный интерфейс

- Создано PWA-приложение (Progressive Web App):
- Push-уведомления о критических событиях
- Быстрый доступ к текущим показаниям
- Управление доступом (открытие двери по PIN)
- Просмотр календаря занятости лаборатории
- Работа в оффлайн-режиме (кэширование последних данных)

#### Задача 4.4: Административная панель

- Отдельный интерфейс для администраторов:
- Управление пользователями и правами
- Конфигурация устройств (пороги, интервалы опроса)
- Настройка правил и автоматизации
- Просмотр системных логов
- Диагностика подключений

**Результат этапа: Система имеет полный набор интерфейсов для всех категорий пользователей, отчётность в реальном времени и за произвольные периоды.**

### Этап 5: Масштабирование и оптимизация

На этом этапе система была подготовлена к работе в промышленном режиме и масштабированию.

#### Задача 5.1: Оптимизация энергопотребления устройств

- Переход на режим глубокого сна (deep sleep) микроконтроллеров:
- Устройства просыпаются каждые 30 секунд для измерения и отправки данных
- При нормальных условиях отправляются только изменения (дельта)
- При тревожных условиях частота увеличивается до 5 секунд

#### Задача 5.2: Обеспечение надёжности

- Реализованы механизмы самодиагностики устройств:
- Проверка целостности датчиков

#### Задача 5.3: Подготовка к масштабированию

- Сервис аутентификации и авторизации
- Сервис управления устройствами
- Сервис обработки данных
- Сервис аналитики и отчётности
- Сервис уведомлений
- Все сервисы контейнеризованы с помощью Docker
- Настроена оркестрация с помощью Docker Compose
- Подготовлены скрипты автоматического развёртывания

#### Задача 5.4: Производительность

- Настроено кэширование частых запросов в Redis
- Реализована балансировка нагрузки для веб-интерфейсов
- Оптимизированы запросы к базам данных (индексы, материализованные представления)
- Настроено CDN для статических файлов веб-интерфейсов

**Результат этапа: Система готова к работе 24/7, устойчива к сбоям, может быть масштабирована на десятки устройств.**
